# Email Templates

Target code: `packages/functions/auth/src/email/templates.rs`

## Owns

- Email verification body.
- Password reset body.
- Default subject lines.
- Optional subject overrides.
- Optional file-based body template overrides.
- Safe interpolation.

## Target Behavior

Templates should be simple and deterministic. They should include only the minimum information needed to verify an email address or reset a password.

The target core ships built-in defaults so a new template deploy can work after Resend is configured. Developers can override the subject lines and HTML body templates through deployment configuration.

Verification links point to an app-owned URL configured by `AUTH_EMAIL_VERIFY_URL_BASE`. The auth Lambda appends the verification token as a `token` query parameter and does not render a verification page itself.

## Runtime Config

```text
AUTH_EMAIL_BRAND_NAME optional
AUTH_EMAIL_SUPPORT_EMAIL optional
AUTH_EMAIL_VERIFY_URL_BASE required for verification emails
AUTH_EMAIL_VERIFY_SUBJECT optional
AUTH_EMAIL_RESET_SUBJECT optional
AUTH_EMAIL_VERIFY_TEMPLATE_PATH optional
AUTH_EMAIL_RESET_TEMPLATE_PATH optional
```

When a template path is absent, the built-in default is used. When a path is configured, startup should load and validate the file once and fail clearly if it is missing or invalid.

Template paths are deployment configuration, not request input. They should point to local files packaged with the Lambda artifact or another explicit deploy-time asset path. Remote template URLs are not part of the target core.

## Supported Variables

Verification template variables:

```text
brand_name
email
verification_url
expires_minutes
support_email
```

Password reset template variables:

```text
brand_name
email
reset_url
expires_minutes
support_email
```

Unknown variables should fail validation instead of rendering as empty strings. The renderer should not support loops, conditionals, includes, arbitrary expressions, or user-provided helper functions.

## Security Invariants

- Escape all user-controlled display values.
- Do not include secrets other than the intended verification or reset link URL.
- Avoid logging rendered templates.
- Links should be short-lived and bound to the stored verification or reset record.
- Verification and reset URLs are generated by auth code, not read from request input.
- Template overrides must not be able to read arbitrary files at request time.
